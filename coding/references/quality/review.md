# 代码审查流程（合并前，按序走）

**0. 明确范围** — 读 PR 描述/变更摘要，搞清"改了什么、为什么、影响面"；diff >1000 行先要求拆分再审（~100 行好、~300 行可接受）。

**1. 先审测试** — 有没有覆盖新行为？测的是**行为**还是 mock 调用次数？修 bug 有没有复现测试？测试不过关，实现先不看。

**2. 过五维**（逐文件 / 逐 hunk）：
- ① **正确性** — 边界 / 空值 / 并发 / 错误路径；行为真符合需求吗。
- ② **可读性** — 命名是否达意；嵌套 >3 层、函数 >50 行 报警。
- ③ **架构** — 职责边界 / 耦合；有没有破坏现有抽象。
- ④ **安全** — 输入验证、注入、鉴权（不止"是谁"还要"有权吗"）、密钥。
- ⑤ **性能** — N+1、未分页列表、无谓 re-render、无尺寸图片。

**3. 逐条写意见** — 格式 `严重度 — file:line` + 问题 + 建议。严重度：🔴`Critical`(阻断) / 🟠`Required`(必改) / 🟡`Nit`(可选) / 🔵`FYI`(信息)。
```
🔴 Critical — auth.ts:42  缺所有权校验，任意登录用户可改他人资源。建议入口加 userId===ownerId。
🟡 Nit — utils.ts:18      三层嵌套三元，建议查表，可读性更好（非阻断）。
```

**4. 死代码确认** — 列出孤立的函数/组件/常量，**问"可删？"再删**，不无声删除。

**5. 依赖审查**（若引入新依赖）— 有无现成方案 → 包大小 → 活跃度 → 漏洞 → 许可证兼容。

**6. 高风险判定** — 涉及不可逆 / 安全 / 跨模块边界 / 难验证的不变量 → 转 `adversarial-review.md`。

**7. 给结论** — ✅通过 / ✅通过(仅剩 Nit) / ❌打回(附必改清单)。**只有 Critical+Required 清零才放行。**
